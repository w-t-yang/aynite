# Internationalization (i18n)

Aynite supports English (`en`) and Chinese (`zh`) with a config-based i18n system designed for both the main renderer shell and runtime-loaded iframe views.

## Architecture Overview

There are two i18n layers:

| Layer | Translations Source | Files |
|-------|-------------------|-------|
| **Main Renderer Shell** | `src/renderer/shared/i18n/translations.ts` | TitleBar, App.tsx, UpdateBanner.tsx, Tile, LayoutVibeModal |
| **Runtime Views** | Each view's `~/.aynite/views/<name>/config.json` under `i18n` key | Settings, and any future views |

## Core i18n Modules

### `src/renderer/shared/i18n/translations.ts`
Flat dictionary of `Record<string, { en: string, zh: string }>` for the main renderer shell strings.
Keys follow a `section.key` naming convention (e.g., `'theme.label'`, `'update.available'`, `'tile.loadView'`).

### `src/renderer/shared/i18n/useI18n.ts`
The `useI18n(locale, customTranslations?)` hook returns a `t(key)` function and the current `locale`.
- `customTranslations` (optional) — allows views to pass their own translations loaded from config.json
- Falls back to English if a key/locale variant is missing
- Falls back to the key itself if no translation entry exists

```typescript
const { t } = useI18n(locale)
<span>{t('theme.label')}</span>  // → "Theme" or "主题"

// Views with custom translations:
const { t } = useI18n(locale, viewTranslations)
```

### `src/renderer/shared/i18n/loadViewI18n.ts`
Helper for views to load hierarchical i18n data from their config.json and flatten it into the format `useI18n` expects.

```typescript
import { loadViewTranslations } from '../../shared/i18n/loadViewI18n'

const customTranslations = useMemo(
  () => loadViewTranslations((viewConfig as any).i18n),
  [],
)
const { t } = useI18n(locale, customTranslations)
```

## How Language Detection Works

### First Load (no existing config)
In `src/main/config/logic.ts`, `detectSystemLanguage()` uses `app.getLocale()`:
- If locale starts with `zh` → default is `'zh'`
- Otherwise → default is `'en'`

### Subsequent Loads
The `language` value is stored in `~/.aynite/config/config.json` under `MainConfig.language`.
On load, if the key is missing/deleted, it falls back to `detectSystemLanguage()`.

### Config Handler
The `'language'` config key is handled by `static-handlers.ts`:
- **get**: Reads from `mainConfig.language`, falls back to `detectSystemLanguage()`
- **set**: Persists to `mainConfig.language` in `~/.aynite/config/config.json`
- Registered in `router.ts` with `staticHandlers`

## Event Flow (RFC-003 Compliant)

When language is switched from the TitleBar menu:

```
1. TitleBar → setLocale('zh')
2. LanguageContext.setLocale → configMutations.set('language', 'zh') via bridge
   ↳ Promise<void> — no local state update, no optimistic set
3. Main process persists to config.json
4. Main process broadcasts LANGUAGE_CHANGED event
5. AppContext single event router receives it
   ↳ calls languageRef.current?.setLocale('zh')
   ↳ relays to all iframe views via postMessage({ type: 'aynite:language-changed' })
6. LanguageContext state updates → TitleBar re-renders with t()
7. Each iframe ViewContext receives the postMessage
   ↳ updates locale state → views re-render with useI18n()
```

### Key Rule: No optimistic state updates
Following RFC-003, the `setLocale` setter only fires the IPC call — it does NOT update local state.
State updates always come through the LANGUAGE_CHANGED event.
This ensures consistency: the event loop is the single source of truth.

## i18n in Main Renderer (`src/renderer/src/`)

All user-facing strings in the main renderer use the `useI18n(locale)` hook.

### Files i18n'd:

| File | What's translated |
|------|------------------|
| `layout/TitleBar.tsx` | Menu items (Settings, Theme, Language, etc.), tooltips, workspace modals |
| `App.tsx` | "Loading Aynite..." |
| `UpdateBanner.tsx` | All update modal strings, badge, buttons. Listens for `LANGUAGE_CHANGED` event. |
| `layout/Tile.tsx` | "Close Tile", "Load View", "Available Views", shortcut labels, loading text, "Tile Options" |
| `layout/LayoutVibeModal.tsx` | Modal title, vibe labels/descriptions, cancel/confirm buttons |

### Pattern:
```typescript
import { useI18n } from '../../shared/i18n/useI18n'

const { locale } = useApp()
const { t } = useI18n(locale)

return <span>{t('some.key')}</span>
```

## i18n in Runtime Views (e.g., Settings Page)

Views loaded at runtime from `~/.aynite/views/<name>/` follow a different approach:
translations live **inside the view's own config.json**.

### View Config Structure
```json
{
  "name": "Settings",
  "description": "Application settings and configuration panel",
  "i18n": {
    "zh": {
      "sidebar": {
        "basic": "基本",
        "appearance": "外观",
        "keybindings": "快捷键"
      },
      "appearance": {
        "title": "外观",
        "description": "使用主题...自定义您的工作区...",
        "presets": {
          "title": "主题预设",
          "description": "选择一个预定义主题..."
        }
      }
    }
  }
}
```

### View Pattern
```typescript
import { useMemo } from 'react'
import { useI18n } from '../../shared/i18n/useI18n'
import { loadViewTranslations } from '../../shared/i18n/loadViewI18n'
import { useView } from '../ViewContext'
import viewConfig from './config.json'

function MyView() {
  const { locale } = useView()

  // Load view-specific translations from config.json (memoized — only once)
  const customTranslations = useMemo(
    () => loadViewTranslations((viewConfig as any).i18n),
    [],
  )
  const { t } = useI18n(locale, customTranslations)

  return <span>{t('my.section.key')}</span>
}
```

### Flattening
The hierarchical `i18n` section gets flattened by `loadViewTranslations()`:
```
{ zh: { sidebar: { basic: "基本" } } }
  → { "sidebar.basic": { en: "sidebar.basic", zh: "基本" } }
```

The `en` fallback defaults to the key name itself, so missing translations show the English key as a fallback.

### Passing `t()` to Child Components
For the Settings page, the `t()` function is passed down via `actions.t`:

```typescript
// Parent (Settings.tsx)
<AppearanceTab
  state={...}
  actions={{
    setThemes: ...,
    t: (key: string) => t(key),
  }}
/>

// Child (AppearanceTab.tsx)
interface AppearanceTabProps {
  actions: {
    t: (key: string) => string
  }
}

export function AppearanceTab({ actions }: AppearanceTabProps) {
  const { t } = actions
  return <span>{t('appearance.title')}</span>
}
```

### Generic Tabs with Prefix
For reusable tab components like `SettingsFolderTab` (used by both Skills and Commands), use a `prefix`:

```typescript
<SettingsFolderTab prefix="skills" t={actions.t} ... />
<SettingsFolderTab prefix="commands" t={actions.t} ... />

// Inside SettingsFolderTab:
const t = (key: string) => _t(`${prefix}.${key}`)
```

## Adding i18n to a New View

1. Add `i18n.zh` section to the view's `config.json` with hierarchical keys
2. Import `useI18n`, `loadViewTranslations`, and `useView`
3. Load translations in a `useMemo` (memoized — only computed once)
4. Create the `t()` function with `useI18n(locale, customTranslations)`
5. Replace all user-facing strings with `t('key')`
6. Pass `t()` down to child components via props

## Translation Keys Naming Convention

```
section.subsection.key
```

Examples:
```
sidebar.appearance
appearance.presets.title
about.update.availableTitle
tile.close
update.available
```

## Tips

- **Chinese translations** should use full-width punctuation （，。？！） for Chinese text
- **`{name}` style placeholders** in translations are replaced using `.replace('{name}', value)` at the call site
- **Color labels** (like "Background", "Primary Accent") are not translated because they represent CSS property names that are consistent across languages
- **Provider names** (Ollama, OpenAI, Anthropic, etc.) are brand names and not translated
- **Technical terms** like "API URL", "Provider" may be kept in English or translated depending on context — use judgment
