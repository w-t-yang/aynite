1. The Config Router Is a God Module

Strong
src/main/config/router.ts — 340 lines, 45+ cases in two switch statements
src/main/config/index.ts — thin pass-through, 50 lines
src/main/config/logic.ts — 250 lines (init + load + save)

Problem

routeGetConfig and routeSetConfig are god functions. Every new feature that needs config goes into one of these switch statements. The router knows about AI sessions, themes, workspace state, git, agents, prompts, skills, commands, tools, view configs, matching views, keybindings — it's a centrifugal force pulling logic from 8+ domains into one file.

Deletion test: if you delete router.ts, complexity doesn't concentrate — it scatters across every IPC handler that currently depends on it. The router is shallow: its interface (a key string + payload) is nearly as complex as its implementation (45+ cases branching to every subsystem).

Solution

Register per-module config handlers via a plugin-like registry pattern. Each domain module (AI, workspace, theme, etc.) registers its own getHandler(key, payload, winId) and setHandler(key, payload, winId) at startup. The router becomes a dispatcher that delegates to registered handlers by key prefix — 'ai-', 'session-', 'theme-', etc.

Benefits

  • Locality: AI config routing lives in src/main/ai/ — change, bugs, knowledge all concentrated.
  • Leverage: The router's interface shrinks from "knows about every domain" to "dispatches by prefix."
  • Testability: Module-level handlers can be unit-tested independently.
  • AI-navigability: An AI agent looking for "where does activeSessionId get set?" lands in src/main/ai/ directly.

2. One File for All Paths (path.ts) Is a Bottleneck

Strong
src/lib/path.ts — 560 lines, 40+ exported functions
Imported by ~20+ files across main, preload, and renderer

Problem

Three completely different responsibilities fused into one file:

  1. Path resolution (40+ getter functions like getAIConfigPath(), getWorkspaceDataPath())
  2. Filesystem I/O (readJson, writeJson, copy, rename, remove, ensureDir)
  3. Security wrappers (secureReadText, secureEditFile, secureGlobSearch)

Solution

Split into three seam-aligned modules:
src/lib/path/resolve.ts — all get*Path() functions (pure, synchronous)
src/lib/path/fs.ts — filesystem I/O wrappers
src/lib/path/secure.ts — domain-validated operations
src/lib/path/index.ts — barrel re-export for backward compat during migration

3. GitService: Monolithic Class with IPC Mixed In

Worth exploring
src/main/git/index.ts — 540 lines, single GitService class
Responsibilities: IPC, status parsing, commit gen, hunk patch, path cache, .git watchers

Problem

Four distinct concerns in one class:

  1. IPC registration — 8+ ipcMain.handle calls in setupIpc()
  2. Git operations — status parsing, porcelain parsing, hunk patching, AI commit generation
  3. .git metadata watching — setupGitWatcher/teardownGitWatcher
  4. Cache management — statusCache, rootCache, debounceTimers

Solution

Extract two seams:
GitCommandRunner — pure git operations (status, diff, commit, parsing). Takes cwd, returns data.
GitWatcherManager — manages fsWatch on .git/HEAD and .git/index. Fires callbacks on change.
CommitGenerator — AI commit message generation (currently ~100 lines buried in IPC)
GitCoordinator — thin class wiring IPC handlers to the above modules.

4. AppContext.tsx: The Renderer's God Component

Worth exploring
src/renderer/src/AppContext.tsx — ~400 lines
25+ state variables, workspace CRUD, layout CRUD, theme, IPC events, updates, tiles, resize, notifications

Problem

AppContext is a God component managing state for workspaces, layouts, tiles, themes, updates, notifications, file switching, window state, settings, debug. 30+ methods through context. Every consumer re-renders on any change. The IPC event handler is a single useEffect with a 15-case switch. The handlersRef pattern is a workaround for the monolithic architecture.

Solution

Compose small contexts:
WorkspaceContext — workspace CRUD, layout switching
ThemeContext — theme list, active theme, color application
UpdateContext — update status, progress, errors
WindowContext — maximized, fullscreen, resize
NotificationContext — active notification
TileContext — active tile, tile controls

5. AI Tools Module Has Proliferated Into Many Concerns

Worth exploring
src/main/ai/tools.ts — 1 file, 14 tool definitions:
Shell execution, task management, memory persistence, file tree walking, workspace config reading

Problem

Each tool in createTools() has grown organically. run_command has 60+ lines of shell spawning with full environment setup. initialize_memory does its own file tree walking. update_task has custom progress counting. Each tool's interface is shallow — a tiny schema hides a large implementation.

Solution

Extract into separate files:
tools/run-command.ts — shell execution, approval
tools/task-manager.ts — create/update/get tasks
tools/memory-manager.ts — initialize/read/update memory
tools/file-operations.ts — read/write/edit/list files
tools.ts stays as a thin registry importing and wiring tool implementations.

6. FileBrowserPage Has Too Many Responsibilities

Speculative
src/renderer/views/file-browser/FileBrowserPage.tsx — ~400 lines
Responsibilities: tab management, file content loading, search, mode selection (fileview/diff/edit/view), git diff, keyboard shortcuts, fs-change debouncing, dataview matching

Problem

FileBrowserPage manages six distinct concerns in one component:
1. Tab bar state (open/close/select files)
2. File content loading + refresh (fs-change events, debouncing)
3. Fileview matching + mode selection (userModeRef pattern)
4. Git diff state management (hasDiff, diffHeadContent, diffRefreshKey)
5. Search bar (query, activeMatchIndex, highlight navigation)
6. Dataview matching for JSON files

The userModeRef pattern (a ref tracking whether the user manually selected a mode, to prevent the async mode-selection effect from overriding their choice) is a symptom of shallow architecture — the effect and the UI controls are fighting each other because they share the same state without a clear protocol.

Solution

Extract into custom hooks and sub-components:
useFileTabs() — tab management (open, close, closeAll, history tracking)
useFileContent(path) — file loading, fs-change refresh, debounce
useFileModes(path) — fileview matching, diff detection, view preview matching, the mode-selection protocol
useSearchBar() — query, match index, navigation
FileSearchBar.tsx — the search UI (currently inline in FileBrowserPage)

Top Recommendation

Start with Candidate #1: the Config Router.

  • Highest impact on AI-navigability — the router is the first place any developer (or AI) looks for "how does config work?" and it's a wall of 45+ switch cases.
  • Lowest risk — a handler registry is a mechanical refactor. Each handler can be extracted one at a time. The old dispatch function can be kept as a fallback during migration.
  • Enables Candidates #3 and #5 — once the registry pattern is in place, the Git module and AI Tools module can register their own config handlers, reducing their coupling to the central config module.
  • Immediate payoff — the next config key you add goes in the feature module, not router.ts. The router stops growing.

After that, Candidate #2 (path.ts split) is a quick win with high leverage — it's purely mechanical and touches nearly every file via improved imports.
