1. The Spells Module Is a God Module (Misnamed)

Strong
src/main/spells/index.ts — IPC wiring + 7 re-exports
src/main/spells/spell-installer.ts — YAML parsing, file walking, config persistence
src/main/spells/skills.ts + commands.ts — listing & restoring
src/main/spells/common.ts — resource path, file finder, error broadcasting

Problem

The spells/ directory has a confused identity. Its name refers to a discontinued concept ("spells") that doesn't appear elsewhere. Yet it owns: skill management, command management, bundled resource path resolution (used by config init, playbook restoration, and every module needing getBundledResourcesPath()), YAML front-matter parsing, recursive file finding, and error broadcasting.

Deletion test: Delete spells/ — complexity reappears in config/logic.ts, main/index.ts, and every module that needs bundled resource paths. The module earns its keep but is named after a dead concept with mixed boundaries.

Solution

Rename and split: skills/ (skill CRUD), commands/ (command CRUD + run IPC), resources/ (getBundledResourcesPath, findFilesRecursively, notifyError), yaml-parser.ts (pure front-matter parsing).

2. RSS & Spotify Hooks Are God Hooks

Strong
src/renderer/views/rss/hooks/useRSS.ts — 350 lines, 18 return values
src/renderer/views/spotify/hooks/useSpotify.ts — 370 lines, 25 return values
Both: data loading, caching, auto-fetch, config mutations, playback/browsing controls

Problem

useRSS.ts (350 lines): Data loading + stale detection + auto-fetch, config CRUD (5 operations), content CRUD (3 operations), filtered queries with multi-source aggregation, panel width persistence.

useSpotify.ts (370 lines): Data loading + stale detection + auto-fetch, authentication (connect/logout), playback controls (7 operations), playlist track loading (cache-first-then-API), 10s polling interval, section/screen state.

The interface is shallow: Each hook returns 18-25 values. Callers can't distinguish state from mutations from computed queries. Testability is low.

Solution

useRSS into 4 hooks: useRSSData, useRSSFeedMutations, useRSSContentMutations, useRSSSelection.
useSpotify into 5 hooks: useSpotifyAuth, useSpotifyData, useSpotifyPlayback, useSpotifyPlaylistTracks, useSpotifyNavigation.

3. RSSApp: God Component with Duplicated Resize Logic

Worth exploring
src/renderer/views/rss/RSSApp.tsx — 330 lines
2x copy-paste resize handlers, 4 modal booleans, 8 callback wrappers

Problem

RSSApp.tsx (330 lines) contains: 2x duplicated resize handlers (~50 lines each, same mousedown/mousemove/mouseup pattern), 4 boolean modal states (could be a single state machine), 8 callback wrappers (thin functions calling hook methods), and conditional rendering logic for singleFeed vs multi-source vs bookmarks.

Duplication also across views: The mousedown/mousemove/mouseup resize pattern exists in both RSSApp and SpotifyApp independently, and TileSplitter uses a different mechanism. A shared useResizablePanel() hook would eliminate ~80 lines from RSS alone.

Solution

Extract useResizablePanel(initialWidth, min, max, onSave) hook. Convert 4 modal booleans to a single ModalStateManager pattern. Inline thin callback wrappers in JSX.

4. Session Operations Scattered Across 4 Files

Worth exploring
src/main/ai/chat.ts — save/load/delete/list sessions
src/main/ai/index.ts — IPC wiring
src/main/config/handlers/workspace-state-handlers.ts — delete via config key
src/renderer/views/workspace-view/WorkspaceView.tsx — session UI + delete logic

Problem

Session lifecycle logic is scattered across 4 locations. The date-directory scanning logic is triplicated within chat.ts itself — loadSession, listSessions, and deleteSession all walk date directories independently with slightly different filtering. The delete endpoint reaches chat.ts through two different paths (IPC handler in ai/index.ts and config key in workspace-state-handlers.ts).

Solution

Create src/main/ai/session-service.ts — a singleton owning all session operations: save, load, list, delete, create. Centralizes date-directory scanning into one method with a filter parameter. The config router and IPC handlers both delegate to it.

5. The Preload Bridge Is a Flat God Object

Speculative
src/preload/index.ts — 250 lines, 1 object, 60+ methods
All domains flat: config, file, workspace, AI, git, system, theme, spells, RSS, Spotify

Problem

aynite.checkIsTextFile sits next to aynite.getGitStatus next to aynite.rssToggleBookmark. No domain isolation. Naming is inconsistent (some prefixed like rssGetConfig, some not like getGitStatus). IPC channels are visible to all renderer code. Every new feature adds more flat methods. env.d.ts must declare the entire flat interface.

Solution

Namespace by domain: aynite.config.get(), aynite.file.read(), aynite.ai.chat(), aynite.git.status(), etc. This is a mechanical rename — IPC channels stay the same, the compiler catches all breaks. Each domain can be a separate module imported by the preload.

window.aynite = {
  config: { get, set },
  file: { list, read, write, save, info, watch },
  workspace: { list, create, delete, switch, folders },
  ai: { chat, prompt, session: { save, load, list, delete } },
  git: { status, refresh, commit, diff, hunk },
  rss: { config, fetch, content, bookmark },
  spotify: { auth, playback, playlists },
  system: { external, fonts, views, dialog },
  events: { on, off, emit },
}

6. Shared Data Lifecycle Pattern Duplicated in RSS & Spotify

Speculative
useRSS.ts — STALE_MS (24h), autoFetchDone ref, loadAll → stale check → fetchAll
useSpotify.ts — STALE_MS (24h), autoFetchDone ref, loadAll → stale check → fetchAll
Identical pattern, independent implementations

Problem

Both hooks follow the identical data lifecycle: 1) load cached data on mount, 2) check staleness (both use 24h), 3) if stale, auto-fetch fresh data, 4) update state. Both use an autoFetchDone ref to prevent duplicate fetches. Both use STALE_MS constants. The pattern is written independently in both modules — a missed abstraction opportunity.

Solution

Extract useStaleData(loadCached, fetchFresh, staleMs, autoFetch) hook to src/renderer/shared/hooks/useStaleData.ts. Takes loadCached() and fetchFresh() callbacks, returns { data, loading, error, refresh }. Reduces both hooks by ~40 lines each and eliminates the duplicated autoFetchDone ref + stale-check logic.

Top Recommendation

Start with Candidate #2: Split the RSS and Spotify god hooks.

  • Highest ROI — Both hooks are 350+ lines with 18-25 return values. Splitting each into 4-5 focused hooks follows the proven FileBrowserPage pattern.
  • Enables #3 and #6 naturally — Once RSS data is separated from RSS mutations, extracting useResizablePanel (for RSSApp) and useStaleData (shared lifecycle) becomes trivial follow-ups.
  • Zero risk pattern — Same mechanical extraction already done for FileBrowserPage. The component trees (RSSApp, SpotifyApp) stay the same; only the hook wiring changes.
  • Immediate testability win — Extracted data hooks can be tested independently with mocked IPC.

After that, Candidate #1 (rename/split spells directory) is a quick win — it resolves the naming confusion and separates skills from commands from infrastructure utilities.
