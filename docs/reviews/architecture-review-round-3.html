1. Settings.tsx: God Component with 10+ Concerns

Strong
src/renderer/views/settings/Settings.tsx — ~450 lines
11 useState calls • 10 tab callback handlers • 9 data domains in one Promise.all
Monolithic restore modal with 7-branch if-else chain • 10 conditional tab renders

Problem

After FileBrowserPage was extracted, Settings.tsx has become the next god component. Every tab handler (handleSetThemes, handleSetAI, handleSetSkills, handleSetMessengers, etc.) lives in the parent, not delegated to the tab. The loadSettings function loads 9 different domains in a single Promise.all — AI config, agents, prompts, keybindings, skills config, skills items, commands config, commands items, tools, and messengers. The restore modal has a 7-case if-else chain for different tab types.

Deletion test: Delete Settings.tsx — complexity doesn't concentrate. It scatters across all tab components (AITab, AgentsTab, SkillsTab, etc.) which currently receive pre-loaded data and would need their own loading logic. The component is shallow: its orchestration interface is nearly as complex as the implementation.

Solution

Adopt a tab-level data loading pattern. Each tab (AITab, AgentsTab, etc.) loads its own data and manages its own saves. Settings.tsx becomes a thin orchestrator handling only tab routing, the sidebar, and the shared restore modal. Tabs receive a lightweight tabApi object with onRestore() and t() rather than pre-loaded state and callback handlers.

Benefits

  • Locality: "How does tools saving work?" — answer is in ToolsTab.tsx, not a handler in Settings.tsx
  • Leverage: Settings.tsx's interface shrinks from 10+ handler props to ~3 (routing + restore)
  • Performance: Tab switching no longer re-runs loadSettings (9 parallel IPC calls)
  • Dead code savings: Tabs that aren't mounted don't make IPC calls at all

2. ChatService: 400-Line Singleton with 7 Fused Concerns

Strong
src/renderer/views/aichat/services/ChatService.ts — 400+ lines
Session state map • Auto-save with 1s timer • Stream dispatch management
Context compaction (AI summarization) • Approval handling • Message send orchestration
Auto-compact threshold checking • Event subscription wiring

Problem

ChatService was extracted from earlier god-module refactoring but has grown to be a monolith in its own right. The compactContext() method alone is 100+ lines containing inline AI summarization — it builds summary messages, makes an AI chat call via the bridge, manages a stream handler with 60-second timeout, writes backup files, and updates metadata. The sendMessage() method is also 100+ lines with embedded command execution, system prompt building, agent loop orchestration, and auto-compact threshold checks.

Duplication: sendMessage() fetches AI config, agent config, tools config, prompts config, workspace folders, active file, and workspace name in a Promise.all — the same data that useAIChat.ts loads independently via loadSettings(). Two callers fetch the same 5+ config keys.

Solution

Extract three seams from ChatService:

  • SessionStore — state Map, subscription management, getOrCreateSession, notify pattern
  • AutoSaver — 1s debounced save, snapshot comparison, metadata building, timer lifecycle
  • CompactService — context compaction algorithm, backup creation, AI summarization call

ChatService stays as the public facade importing and composing the above services, with sendMessage staying as the orchestration entry point.

3. window.ts: Main Process God Module

Worth exploring
src/main/window.ts — 300+ lines
src/main/window-state.ts — per-window state registry
5 distinct concerns fused: window lifecycle • approval queue • IPC routing • dialogs • permissions

Problem

window.ts has accumulated responsibilities beyond "create and manage windows":

  1. Window lifecycle — createMainWindow, createNewWindow, minimize/maximize/close, onBeforeInputEvent (15+ functions)
  2. Approval queue — requestAiApproval, pendingApprovals Map, approvalQueue, processApprovalQueue, setupAiListeners with ipcMain.on(AiEventChannels.APPROVAL_RESPONSE) (50+ lines of AI-domain logic)
  3. IPC routing — sendToWindow (scoped), broadcastAppEvent (global), sendAppEvent (deprecated), sendAppOperation, sendOperationToWindow, getWinIdFromSender, toggleDevToolsForSender (10+ functions)
  4. Dialogs — showOpenDialog, showSaveDialog

The approval queue is the most intrusive — AI approval logic with queuing, pending map, and event listeners lives in window.ts because it needs BrowserWindow.webContents to send events. But the discipline of "only window.ts touches BrowserWindow" has pushed AI-domain code into window.ts rather than into a separate module that imports a send function.

Solution

Extract two seams:

  • src/main/approval-queue.ts — approval request queue, pendingApprovals Map, processApprovalQueue, AiEventChannels listener setup. Imports sendToWindow from window.ts.
  • src/main/ipc-utils.ts — sendToWindow, broadcastAppEvent, sendOperationToWindow, getWinIdFromSender, toggleDevToolsForSender. This becomes the "only touches BrowserWindow" module.

window.ts retains: createMainWindow, createNewWindow, window state helpers, permission handling, window event listeners.

4. Telemetry Module: Singleton with 4 Concerns

Worth exploring
src/main/telemetry/index.ts — 250+ lines
GA4 protocol client • Buffer management • Timer lifecycle (flush + heartbeat) • Config persistence

Problem

The telemetry module owns four distinct concerns in one file:

  1. GA4 protocol client — GA_URL, GA_API_SECRET, fetch calls, event serialization, user_properties
  2. Buffer management — in-memory event buffer, flush-on-25-items, forced flush on quit
  3. Timer lifecycle — flush timer (60s), engagement heartbeat (30s), start/stop for each. The start/stop timer pattern is duplicated between both timers.
  4. Config persistence — loadTelemetryConfig, saveTelemetryValue, first_open tracking

Duplicated timer pattern: The flush timer and heartbeat timer share the same lifecycle (start → stop → cleanup), but implemented independently. A shared PeriodicRunner utility would eliminate ~15 lines of duplicated interval management.

Solution

Extract a shared PeriodicRunner utility class for the main process (wraps setInterval with start/stop/cleanup). Then split telemetry into: ga4-client.ts (GA4 protocol, fetch, event serialization) and index.ts (session lifecycle, config persistence, event tracking API).

5. agent.ts: Core Orchestrator Living in utils/

Worth exploring
src/renderer/views/aichat/utils/agent.ts — 100+ lines, core agent loop
src/renderer/views/aichat/utils/message.ts — pure message helpers
src/renderer/views/aichat/utils/commands.ts — command execution
src/renderer/views/aichat/utils/input.ts — input serialization

Problem

runAgentLoop() is the core AI agent orchestrator — it manages message accumulation, tool call tracking, assistant message flushing, real-time command output streaming, and the full lifecycle of an agent interaction. But it lives in a utils/ directory alongside message.ts (pure message helpers) and input.ts (input serialization).

Apply the naming test: Does "utils" tell a developer what's inside? No. It's a miscellany — four files with four fundamentally different responsibilities (orchestration, message manipulation, command execution, input parsing) bundled under one label.

Solution

Promote agent.ts to a first-class module at src/renderer/views/aichat/agent-loop/index.ts. The utils/ directory keeps message.ts, commands.ts, input.ts — actual utilities. No code changes needed, only a directory rename and import path update.

6. Duplicated Header Toolbar Pattern Across Views

Speculative
RSSApp.tsx — header with icon + title + RefreshCw + tooltip (~30 lines)
SpotifyApp.tsx — header with icon + title + RefreshCw + LogOut (~35 lines)
AIChat.tsx — Header component with history, clear, copy (~40 lines)
FileBrowserPage.tsx — TabBar + StatusBar (different pattern)

Problem

Every view has a header toolbar with common interaction patterns — icon + title on the left, action buttons with tooltips on the right, loading spinners on refresh buttons. RSS, Spotify, and AI Chat all implement slightly different versions. The common structure:

  • Left section: icon + view title
  • Right section: action buttons with tooltips
  • Consistent hover/disabled/loading states

This is speculative because the duplication is small (~30 lines each) and a generic ViewToolbar component might end up with too many props. Worth raising but not a priority.

Solution

A small ViewToolbar component at src/renderer/shared/basic/ViewToolbar.tsx with a simple slot-based API:

<ViewToolbar icon={<Rss size={14} />} title="RSS Reader">
  <button onClick={rss.fetchAll} title="Refresh"><RefreshCw /></button>
</ViewToolbar>

Top Recommendation

Start with Candidate #1: Settings.tsx — push data loading into tab components.

  • Highest impact on navigability — Settings is the first place any developer goes to understand how config works. Moving data loading into each tab makes it immediately clear which tab owns which config domain.
  • Lowest risk — Each tab component already exists and already has access to the bridge. The change is purely about moving the config.get/set calls from the parent into the child. Each tab can be migrated independently.
  • Enables #5 naturally — Once tabs are self-loading, the restore modal can be simplified (each tab exposes its own restore handler instead of a 7-branch if-else).
  • Immediate performance win — The monolithic loadSettings fires 9 parallel IPC calls on every tab switch. Tab-level loading fires only the calls needed for that tab.

After that, Candidate #2 (ChatService extraction) has the highest return on testability — the extracted AutoSaver, SessionStore, and CompactService would each be independently testable. Candidate #3 (window.ts extraction) is a good follow-up that cleans up the main process boundary.
